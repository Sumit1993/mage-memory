# 0006 — Two-layer recall: per-wing auto-loaded skills + a hierarchical factual index

Recall uses two complementary layers:
1. **Skills** — the procedural/nudge/capability layer: auto-loaded at session start across all skill-capable agents (portable), model-matched to the task, executable. = the lifecycle skills (`learn`, `dream`, awareness) **plus one skill per wing** that is the wing's auto-loaded entry point (its playbooks/gotchas/wayfinding, with details in referenced files via progressive disclosure).
2. **Hierarchical index** — the factual/pointer layer: a small always-loaded root `INDEX.md` (lists wings) + per-wing index files loaded on demand; scalable, navigated (pull), regenerated by `mage index`.

## Decision
- **One skill per wing, auto-loaded** — a flat, simple rule (no conditionality). Lifecycle skills: `learn`, `dream`, awareness.
- **Hierarchical index** for the many factual notes (root → per-wing).
- **No automated note↔skill promotion/demotion engine in v1** — judged overreaching at this stage. The promotion ladder (and harvesting the homunculus instinct→skill algorithm, ADR-0005) is **deferred** until a bounding need actually emerges (e.g., wings proliferating).

## Consequences
- Skill catalog is bounded by a simple rule (lifecycle + one-per-wing), not an engine — simpler to build and reason about.
- A wing's gotchas/playbooks live in its wing skill (sections/referenced files), so they auto-surface when working that wing — the "nudge," authored directly.
- `/dream` maintains notes (staleness/supersede/consolidate); it does **not** manage skills in v1.
- Revisit if a system accrues many wings (always-loaded skill metadata grows) — then reintroduce promotion/bounding.
