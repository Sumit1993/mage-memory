# mage ‚Äî context & glossary

mage is a portable, file-based, self-maintaining knowledge base for software systems ‚Äî durable git-backed **notes** that resist accidental loss, navigable as an Obsidian **graph**, usable by any AI coding agent. It exists because project knowledge (specs, decisions, how services connect, how to call them) scatters across in-repo/out-of-repo dirs and gets cleaned up or lost; mage gives it one durable, discoverable, portable home. mage stores **insight, procedure, and pointers** ‚Äî how to act and where to look, so you do it faster next time ‚Äî *not* copies of sources that already exist.

## Language

### Structure

**knowledge base**:
The whole memory for a system ‚Äî the graph of notes plus its index. One per repo (in-repo) or one hub spanning many repos (the system "brain").
_Avoid_: docs, wiki, vault (as the product noun)

**Note**:
The atomic unit ‚Äî one verbatim markdown file about one thing, with frontmatter (type, tags, provenance, lifecycle) and portable links to related notes.
_Avoid_: drawer, doc, memory (as a noun), entry

**Wing**:
The top-level scope a note belongs to ‚Äî typically a project, repo, service, or person. Realized as a nested tag (`#wing/<name>`) and/or a MOC, **not** a mandatory folder. A wing is an **optional convention, never a necessity**: untagged notes are valid (they index as *Cross-cutting*) ‚Äî reach for a wing only when a base spans more than one top-level scope. The **first** tag is a note's *primary* wing (drives color + ownership); a note may carry more tags and is indexed under **each** of them (multi-home). In a hub, the whole system is itself a wing (named after the hub). See [ADR-0011](../decisions/0011-recursive-scan-hub-projects.md), [ADR-0012](../decisions/0012-wings-optional-convention-standalone-hubs.md).
_Avoid_: section, bucket

**Room**:
A topic within a wing ‚Äî the second scoping level. Realized as a nested tag (`#wing/room`) and/or a topic MOC.
_Avoid_: category, folder

**Index**:
The compact, always-available pointer layer ‚Äî one line per note (topic ¬∑ keywords ¬∑ ‚Üí link), generated from the notes. An agent loads this to know what exists and decide what to open. (The "closet" of the memory palace.)
_Avoid_: closet, manifest, catalog

**Tag**:
A `#wing/room`-style nested label for emergent grouping. Tags carry the hierarchy; folders are avoided until a cluster is large and stable.

**MOC** (Map of Content):
A note whose body links a cluster of related notes ‚Äî an emergent, optional navigation hub, added when a wing/room earns enough notes to deserve a map.

**Skill**:
The procedural/auto-loaded counterpart to a Note ‚Äî a capability the agent loads when relevant (portable across skill-capable agents), carrying procedures/playbooks/gotchas and able to run scripts. Notes are recalled by navigation (pull); skills are pushed (auto-loaded). mage keeps three kinds:
- **Lifecycle skill** ‚Äî a mage capability itself (`learn`, `dream`, ‚Ä¶).
- **Wing skill** ‚Äî one auto-loaded entry point per Wing (points at the wing's index + surfaces its playbooks/gotchas). Generated by `mage skills`.
- **Procedure skill** ‚Äî a single procedural note (a Playbook/Gotcha) **graduated** to its own individually-loadable SKILL.md with its own trigger, so it can be redacted, usage-tracked, grouped, and optimized as one addressable artifact. The note stays the substrate; the skill is its pushed form ([ADR-0006](../decisions/0006-two-layer-recall-per-wing-skills.md)).
_Avoid_: command (as a synonym)

### Work & artifacts

**Work unit**:
A task-scoped container (`work/<slug>/`, open `type`: spec, investigation, incident, spike, ‚Ä¶) holding the working notes for one piece of work ‚Äî the *lab notebook* where work happens before durable knowledge is distilled into notes. Tracked in git; its `artifacts/` are not.
_Avoid_: ticket, task-folder

**Artifact**:
A generated or downloaded working material (PDF, profile, query output, repro script) in a work unit's `artifacts/`. Durable on disk ‚Äî **never `/tmp`** ‚Äî and citable from notes as evidence. An artifact is raw material; a note is distilled knowledge.
_Avoid_: attachment, temp file

### Note types (`type` frontmatter ‚Äî a *suggested, open* vocabulary, never enforced)

**Spec / Plan / Tasks**: greenfield intent authored forward (the SDD path).
**Decision**: an ADR ‚Äî a choice and why, and what it rules out.
**Principle**: a governing constraint (the constitution).
**Interface**: how to *use* a service/API ‚Äî endpoints, useful params, auth, gotchas (verbatim).
**Tooling**: how a repo is wired to external tools ‚Äî VCS, CI, security analysis, deploy, monitoring (a repo‚Üîtool relationship).
**Topology**: architecture(s), deployment methods, environments across the system.
**Relationship**: a *governed* coupling between wings/services/repos worth tracking ‚Äî carries `breaks_on`, contract-anchors, owners, review cadence. Lightweight ties are just links (edges); a Relationship note is a reified node.
**Playbook**: how to do X faster ‚Äî a reusable procedure.
**Gotcha**: what *not* to do and why (e.g., a CLI flag that fails) ‚Äî surfaced to prevent the repeat mistake.
**Pointer / Reference**: where a canonical source lives and when to go there (`sources:` ‚Äî URL ¬∑ ticket ¬∑ `file:line`). A wayfinding note, never a copy.
**Trail**: the path that connected sources into understanding (ticket ‚Üí PR ‚Üí code ‚Üí doc).
_Avoid_: cross-ref (as a directory)

**Source**:
A pointer to canonical external knowledge (URL, ticket, `file:line`) carried in a note's `sources:` ‚Äî referenced, never copied. Snapshot to `artifacts/` only when the source itself is fragile.

### Lifecycle

**Remember / Recall**: write to / read from the knowledge base.
**Learn**: capture a new note from work in progress (the brownfield path) ‚Äî draft ‚Üí overlap-check ‚Üí human-confirm ‚Üí promote.

**Distill**: the auto-capture sibling of Learn ‚Äî read mage's *own* observed scratch (`.learnings/`) and propose durable notes from it, **on first sight** (no recurrence gate ‚Äî that is 0.0.8). A deterministic `mage distill --json` reader groups un-distilled events into candidate clusters (chunked by `compact`/session boundary, salience-filtered) under a per-session **offset watermark** in `.metrics/distill.json` (advanced explicitly via `mage distill --seen` after the human dispositions a batch); the `mage:distill` judgment skill reads them through four lenses ‚Äî **user corrections & nudges** (first-class), error‚Üífix, repeated workflow, tool preference ‚Äî and drafts notes through Learn's shared pipeline (overlap-check ‚Üí redact Gate 2 ‚Üí human-confirm ‚Üí write). Reads only mage's own artifacts; foreign memory stores are **ignored, not harvested**. See [ADR-0018](../decisions/0018-mage-distill-observed-scratch-reader.md).

**Dream**: the maintenance pass that keeps memory latest+correct ‚Äî flags staleness, supersedes contradictions, **merges** (consolidates) thin same-topic notes, **splits** notes that grew too big / went incoherent / hide a self-recurring slice, archives, and re-verifies notes against changed source. It is the single **Applier** of all note + skill mutations. Nothing is hard-deleted (git history + `archive/`). See [ADR-0019](../decisions/0019-mage-promote-self-grooming.md) (merge/split).

**Promote**: move up the durability ladder, each rung gated by **recurrence** ‚Äî counted as **distinct sessions** a pattern reappears in, via a deterministic `(wing+tags)`-signature tally that survives the raw-event purge ([ADR-0019](../decisions/0019-mage-promote-self-grooming.md)). Two rungs: *scratch ‚Üí note* (a pattern that recurs in ‚âà3 sessions with no covering note becomes a draft note ‚Äî the catch-net behind Distill's first-sight pass) and *note ‚Üí Procedure skill* (**Graduate** ‚Äî a **Playbook/Gotcha** note corroborated across ‚âà5 sessions earns its own loadable skill). Only *procedural* notes graduate (you auto-load a procedure, not a fact); recurrence gates graduation, **context-match** gates the reword/demote that follow. All proposals are human-confirmed (Rung A); the confirm is the commit.

**Demote**: the reverse of the note ‚Üí skill rung ‚Äî a Procedure skill that stops earning its auto-load falls back to a plain note (never hard-deleted; git history + `archive/`). The signal that drives demotion is usage + **context-match**.

**Redact**: strip secrets and PII at every write that could escape the machine ‚Äî a fast regex scrub at capture (`mage observe` ‚Üí the gitignored `.learnings/`, ECC parity) and a stronger deterministic scan + agent judgment-strip at the **commit boundary** (note/skill creation), where a miss becomes shared. Principle: *redact before write, not before display* ‚Äî block promotion on a likely live secret.

**Context-match**: the deterministic signal of whether a Procedure skill fired in a *fitting* situation ‚Äî when it auto-loads, did the work that followed actually touch its wing/keywords/paths? Computed over a bounded forward window after each `skill_load`, scored as `matched_loads / total_loads` keyed on the trigger's `trigger_hash`. Persistently low match across many loads means the skill's trigger selects the wrong scenarios and the trigger should be reworded (or the skill demoted). Paired with an optional agent self-report ("this skill helped / was irrelevant"), it is the evidence the optimize step gates on ‚Äî mage's no-benchmark, no-core-model stand-in for SkillOpt's held-out selection score. See [ADR-0015](../decisions/0015-mage-observe-capture-schema.md) (the schema it reads), [ADR-0016](../decisions/0016-context-match-confidence-ladder-applier.md) (how it computes).

**Observe event**: one line in `.learnings/<session>.jsonl`, written by the `mage observe` plumbing command from a host hook. A versioned, discriminated record (`{v, ts, session, type, ‚Ä¶}`); the critical-path types are `session_start`, `user_prompt`, `skill_load`, `tool_use`, `compact`, `session_end`, and `assistant_msg` (the agent's final reply per turn, redacted ‚Äî added in [ADR-0019](../decisions/0019-mage-promote-self-grooming.md) to give a correction its antecedent). `tool_use` is a *salient extract* (structured `paths` + a bounded `detail`), never a transcript copy; `skill_load` snapshots the trigger's `match` spec + `trigger_hash` at load time. Gate-1 redaction scrubs free-text fields before write. The schema is load-bearing for the whole self-grooming loop ([ADR-0015](../decisions/0015-mage-observe-capture-schema.md)).
_Avoid_: log line, observation (as the product noun)

**Rollup**: the derived, gitignored, purge-exempt tally of context-match per `(skill, trigger_hash)` in `mage/.metrics/`. Persists after raw observe events purge; never committed. The principle: *metrics never enter git ‚Äî only the human-confirmed mutations they motivate do.*

**Recurrence tally**: Promote's counter ‚Äî a second deterministic fold (Distill's sibling) over the same `.learnings/` scratch, keyed on a `(wing+tags)` **signature** and counting **distinct sessions**. Gitignored `mage/.metrics/promote.json` with its own bookmark; purge-surviving and fold-based (never backfilled from raw events). The coarse signature is *bucketed* deterministically and *refined* by judgment at proposal time ([ADR-0019](../decisions/0019-mage-promote-self-grooming.md)).

**Proposal queue / Rejected-edit buffer**: the gitignored `mage/.metrics/` queues the detection stages write into ‚Äî `proposals.json` (pending: note? ¬∑ graduate? ¬∑ merge? ¬∑ split? ¬∑ reword?) and `rejected.json` (what the human declined, so mage backs off). The rejected buffer is the "back off" half of the **accept/reject ladder** that adapts mage's eagerness to each user without persona detection ([ADR-0016 ¬ß3](../decisions/0016-context-match-confidence-ladder-applier.md)).

**Sensitivity dial**: the single human knob (`low | normal | high`, default `normal`) that scales Promote's thresholds together ‚Äî a *choice*, so it lives in the **tracked** KB config (`metadata.json`), unlike the gitignored counters. The default numbers live in code (the **seam**); auto-tuning them from usage is a deferred opt-in dream rung that would key on accept/reject, never raw volume or a guessed persona ([ADR-0019](../decisions/0019-mage-promote-self-grooming.md)).

**Applier**: the single piece of code ("dream") that mutates the skill catalog (graduate / demote / reword) and notes (supersede / consolidate-merge / **split** / prune / archive). Detection (promote/optimize) only *proposes*; the applier is the one serialized writer and the choke point that enforces the never-loosen ceilings (no auto-commit ¬∑ no bespoke-skill edit ¬∑ no hard-delete ¬∑ no past-Gate-2). See [ADR-0016 ¬ß4](../decisions/0016-context-match-confidence-ladder-applier.md).

**Plumbing command vs Human verb**: mage's CLI splits in two. *Plumbing seams* (`observe`, `redact --check`, `ingest --json`, `distill --json`) are invoked by **machinery** ‚Äî host hooks, the graduate skill, a git pre-commit hook ‚Äî never typed by humans; they exist only because hooks/skills/git reach mage through the CLI. *Human verbs* (`init`, `index`, `skills`, `dream`, `learn`, `link`, `list`, `status`, `doctor`) are what people type. A hook may *fire* a deterministic plumbing command or *nudge* a judgment verb, but never reasons itself ([ADR-0009](../decisions/0009-no-runtime-automation-rides-host-hooks.md)). See [CONVENTIONS ¬ß10](../../CONVENTIONS.md).

**Connect / Disconnect**: the opt-in act of wiring (or removing) mage's capture into a host. `mage connect` writes the `mage observe` hook block ‚Äî `id:"mage:*"`-marked, idempotent ‚Äî into `.claude/settings.local.json` (per-repo, gitignored; `--user` for all sessions); `mage disconnect` removes only those entries. Capture is **opt-in, never bundled with the skills plugin**, and `connect` **fully ignores** any other observer (e.g. ECC) ‚Äî they coexist; mage reads only its **own** artifacts and ignores foreign memory stores entirely (no harvest, no feeder ‚Äî [ADR-0018](../decisions/0018-mage-distill-observed-scratch-reader.md)). The per-harness adapter ([ADR-0009](../decisions/0009-no-runtime-automation-rides-host-hooks.md), [ADR-0017](../decisions/0017-mage-connect-host-hook-adapter.md)); the `.learnings/` schema is harness-neutral so other harnesses are additive input adapters.
_Avoid_: install/uninstall (as the product noun)
**Supersede**: replace a note's claim with a newer one, keeping the old linked + marked (never a silent overwrite).

## Flagged ambiguities

- **Wing vs Project** *(resolved, [ADR-0011](../decisions/0011-recursive-scan-hub-projects.md))*: a **hub is one vault**; a **project** is a sub-scope of it whose notes live under `projects/<name>/` and surface as a **wing** (`<name>/room`), plus a **registry entry** in the hub `metadata.json` recording its code repo. So a project *is* a wing (how it groups + indexes ‚Äî tag-derived) **and** a registry entry (how it's labelled/linked). A wing need not be a project ‚Äî it can be a non-repo scope (a person, an external service) or hub-level cross-cutting notes.
- **"Memory" the quality vs the noun**: knowledge has *durable memory*; we don't call a note "a memory."

## Example dialogue

**Dev:** I just figured out the payments service double-bills unless every charge carries an idempotency key.

**mage:** That's an **Interface** note about payments. Which **wing** ‚Äî is `payments` its own wing, or a room under `billing`?

**Dev:** Room under `billing`. And it's tied to how the `web` repo calls it.

**mage:** Then the idempotency fact is an Interface note tagged `#billing/payments`, and the `web ‚Üí payments` coupling is a **Relationship** note with `breaks_on: charge-schema, missing idempotency header`. I'll check the **index** for existing payments notes first, draft both, and ask you to confirm before they're promoted. The relationship becomes a graph edge between the `web` and `billing` wings.
